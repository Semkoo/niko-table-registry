Drag any header's right edge to resize that column, or double-click the grip to autosize to the widest visible content. Drop `<DataTableColumnResize />` inside `DataTableRoot` to enable it: columns keep working with regular `DataTableHeader` / `DataTableBody` (no virtualization required). Opt out per column with `enableResizing: false`.

Columns default to a **40px** drag floor (`minSize`) when resize is enabled: override per column with `minSize` / `maxSize`. Body cells use `truncate` so long text ellipsizes instead of spilling into the next column (same as the Data Grid / virtualized bodies).

**Fill is on by default.** With resize enabled the table stretches to its container and one **flex column** absorbs the leftover width, so a trailing actions column pins to the right edge instead of leaving dead space. The flex column is the first non-pinned data column by default: point it at another with `meta: { flex: true }`, opt a column out with `meta: { flex: false }`, or turn filling off for the whole table with `meta: { disableFlexFill: true }` (`TableMeta`). It's pure layout, never written to `columnSizing`. Un-resized columns are also floored at their **header width** (header-fit) so labels never truncate on load. The flex column stays **resizable** too: dragging its grip pins it at an explicit width and hands the fill to the next eligible column.

Prefer to spread the surplus across every resizable column instead of one flex column? Add **`<DataTableColumnAutoFit />`** to scale them equally on load, backing off once you drag a column or restore a saved layout. See [Auto-fit](#auto-fit) and [Persist Widths](#persist-widths).

## Introduction

Column resizing lets users tweak widths for dense, multi-column tables: especially when content length varies (emails, notes, product names). Explicit `size` values make the initial layout predictable; the resize addon handles drag and double-click autosize.

## Enable Column Resize

Drop `<DataTableColumnResize />` early inside `DataTableRoot`. Its presence flips `enableColumnResizing` via feature detection: no config flag required.

```tsx showLineNumbers title="column-resize.tsx"
<DataTableRoot data={data} columns={columns}>
  <DataTableColumnResize />
  <DataTable>
    <DataTableHeader />
    <DataTableBody />
  </DataTable>
</DataTableRoot>
```

## Column Definitions

Set explicit `size` on columns so the initial layout is meaningful and horizontal scroll appears when needed. Opt out of resizing on fixed columns (e.g. actions):

```tsx showLineNumbers title="columns.tsx"
const columns: DataTableColumnDef<Order>[] = [
  {
    accessorKey: "id",
    size: 110,
    header: () => (
      <DataTableColumnHeader>
        <DataTableColumnTitle title="Order ID" />
        <DataTableColumnActions>
          <DataTableColumnSortOptions />
        </DataTableColumnActions>
      </DataTableColumnHeader>
    ),
    meta: { label: "Order ID" },
  },
  {
    accessorKey: "customer",
    size: 160,
    header: () => (
      <DataTableColumnHeader>
        <DataTableColumnTitle title="Customer" />
        <DataTableColumnActions>
          <DataTableColumnSortOptions />
        </DataTableColumnActions>
      </DataTableColumnHeader>
    ),
    meta: { label: "Customer" },
  },
  // ... more columns with explicit sizes
  {
    id: "actions",
    size: 70,
    enableResizing: false, // no resize grip on this column
    enableSorting: false,
    enableHiding: false,
    // ...
  },
]
```

## Auto-fit

By default a single flex column absorbs the leftover width. When you'd rather distribute the surplus evenly across every resizable column, drop **`<DataTableColumnAutoFit />`** next to `<DataTableColumnResize />` and the **resizable** columns scale up to fill the container on load. It's a self-contained, opt-in feature: without it, columns keep their declared `size` and none of the auto-fit code ships. Fixed columns (`enableResizing: false`) keep their size; auto-fit re-fits when the container grows and backs off once the user drags a column or a saved `columnSizing` is restored, so manual and persisted widths win. Columns wider than the container scroll horizontally.

Install it alongside resize:

<CliCommandCode action="run" command="shadcn@latest add @niko-table/data-table-column-auto-fit" />

```tsx showLineNumbers title="auto-fit.tsx"
<DataTableRoot data={data} columns={columns}>
  <DataTableColumnResize />
  <DataTableColumnAutoFit />
  <DataTable>
    <DataTableHeader />
    <DataTableBody />
  </DataTable>
</DataTableRoot>
```

## Controlled Sizing

Manage `columnSizing` externally for persistence or live inspection. Provide `state={{ columnSizing }}` + `onColumnSizingChange`. Restored widths take precedence over auto-fit, so a saved layout is never overwritten on reload:

```tsx showLineNumbers title="column-resize-state.tsx"
import { useState } from "react"
import { type ColumnSizingState } from "@tanstack/react-table"

export function ControlledResizeTable({ data }: { data: Order[] }) {
  const [columnSizing, setColumnSizing] = useState<ColumnSizingState>({})

  return (
    <DataTableRoot
      data={data}
      columns={columns}
      state={{ columnSizing }}
      onColumnSizingChange={setColumnSizing}
    >
      <DataTableColumnResize />
      {/* ... */}
    </DataTableRoot>
  )
}
```

## Persist Widths

Want resized widths to survive a reload? Install the opt-in **`useColumnSizingPersistence`** hook and wire its `columnSizing` + `onColumnSizingChange` into `DataTableRoot`. It's `localStorage`-backed, writes immediately, is SSR-safe, and ignores corrupt storage. Resize a column below, then reload:

<CodePreview demo="niko-table/column-resize-persisted" />

<CliCommandCode action="run" command="shadcn@latest add @niko-table/data-table-column-sizing-persistence" />

```tsx showLineNumbers title="orders-table.tsx"
import { useColumnSizingPersistence } from "@/components/niko-table/lib/use-column-sizing-persistence"

export function OrdersTable({ rows, orgId }) {
  // Scope the key per table (and per tenant/user if widths shouldn't be shared).
  const { columnSizing, onColumnSizingChange } = useColumnSizingPersistence(
    `orders-table:${orgId}`,
  )

  return (
    <DataTableRoot
      data={rows}
      columns={columns}
      state={{ columnSizing }}
      onColumnSizingChange={onColumnSizingChange}
    >
      <DataTable>
        <DataTableHeader />
        <DataTableBody />
      </DataTable>
      <DataTableColumnResize />
    </DataTableRoot>
  )
}
```

The hook also returns `resetColumnSizing()` to clear the saved widths. Prefer to own the storage yourself (e.g. sync widths through a per-user settings API instead of `localStorage`)? It's a thin wrapper — copy it and swap the store; all `DataTableRoot` needs is a controlled `columnSizing` + `onColumnSizingChange`.

> Persist column **width** like visibility and order: a per-user preference, not URL state. Restored widths take precedence over auto-fit, so a saved layout is never overwritten.

## When to Use

**✅ Use Column Resize when:**

- Tables have many columns and users need to adjust widths to content
- You want drag-to-resize and double-click autosize without a custom header
- Some columns (row number, actions) should stay fixed-width via `enableResizing: false`

**Works with:**

- [Row DnD](/examples/row-dnd-table/): same `<DataTableColumnResize />` on `DataTableHeader` + `DataTableDndBody` (opt out on drag-handle columns)

**❌ Consider other options when:**

- All columns fit comfortably and widths never need tuning
- You only need sticky identifier/action columns: see [Column Pinning](/examples/column-pinning-table/)

## Virtualization

You do **not** need a separate “virtualized column resize” example. Swap in `DataTableVirtualizedHeader` + `DataTableVirtualizedBody` (and give `DataTable` a fixed height): the same `<DataTableColumnResize />` marker applies.
