The grid ships six cell editors, each following the same **display / editor split**: a cheap read-only shell renders for every visible cell, and the heavy editor mounts only for the one cell being edited — so a 280-cell viewport never constructs 280 date pickers. The grid stays type-agnostic; a cell's validity comes from the `resolve` you inject.

## The built-in editors

| Cell     | Component                             | Source                          | Notes                                                                                  |
| -------- | ------------------------------------- | ------------------------------- | -------------------------------------------------------------------------------------- |
| Text     | `GridTextCell`                        | `grid/cells/grid-text-cell`     | Uncontrolled input; commits on blur / Enter / Tab.                                     |
| Number   | `GridNumberCell`                      | `grid/cells/grid-text-cell`     | Right-aligned; `resolve` decides numeric validity.                                     |
| Currency | `GridNumberCell` + `format`           | `grid/cells/grid-text-cell`     | Display formats via `Intl.NumberFormat`; the editor still shows raw for precise entry. |
| Checkbox | `GridCheckboxCell`                    | `grid/cells/grid-checkbox-cell` | Always interactive; Space / type toggles it.                                           |
| Date     | `GridDateCell`                        | `grid/cells/grid-date-cell`     | Popover calendar; stores `YYYY-MM-DD` (no timezone shift).                             |
| Select   | `GridSelectCell` / `GridComboboxCell` | `grid/cells/grid-select-cell`   | Command + Popover dropdown; `GridComboboxCell` adds search.                            |

## Dispatching per column

A consumer picks the editor per column — this palette is yours, the grid never sees it. A tagged column spec + a small dispatcher is the common shape:

```tsx showLineNumbers
function CellByType({
  spec,
  ...p
}: CellEditorProps & { spec: ColumnSpec }) {
  switch (spec.type) {
    case "number":
      return <GridNumberCell {...p} resolve={(raw) => resolveCell(spec.id, raw)} />
    case "currency":
      return (
        <GridNumberCell
          {...p}
          resolve={(raw) => resolveCell(spec.id, raw)}
          format={(raw) => usd.format(Number(raw))}
        />
      )
    case "checkbox":
      return <GridCheckboxCell {...p} aria-label={spec.label} />
    case "date":
      return <GridDateCell {...p} placeholder={spec.label} />
    case "select":
      return (
        <GridSelectCell
          {...p}
          options={spec.options.map((o) => ({ label: o, value: o }))}
          placeholder="Select…"
          displayLabel={p.cell.value ?? undefined}
        />
      )
    default:
      return <GridTextCell {...p} resolve={(raw) => resolveCell(spec.id, raw)} />
  }
}
```

## Validity via `resolve`

Number, date, and select cells stay valid/invalid based on the `resolve` you pass — the grid stores the `status` but never computes it. A pasted-but-unmatched value stays visible (in red) so the user can correct it:

```tsx showLineNumbers
function resolveCell(columnId: string, raw: string): CellState<string> {
  const spec = SPEC_BY_ID[columnId]
  const trimmed = raw.trim()
  switch (spec?.type) {
    case "number":
    case "currency": {
      if (trimmed === "") return { raw, value: null, status: "empty" }
      const ok = Number.isFinite(Number(trimmed))
      return { raw, value: ok ? trimmed : null, status: ok ? "valid" : "invalid" }
    }
    case "select": {
      const match = spec.options.includes(raw)
      return {
        raw,
        value: match ? raw : null,
        status: raw === "" ? "empty" : match ? "valid" : "invalid",
      }
    }
    default:
      return { raw, value: raw || null, status: raw === "" ? "empty" : "valid" }
  }
}
```

## Writing your own cell

Any component satisfying the `CellEditorProps` contract is a cell — so a rating stepper, a color swatch, or a multi-select drops in the same way. Follow the split: render `GridCellDisplay` while not editing, and mount your editor only when `isEditing`.

```tsx showLineNumbers
import { GridCellDisplay } from "@/components/niko-table/grid/cells/grid-cell-display"
import type { CellEditorProps } from "@/components/niko-table/grid/cells/cell-props"

export function MyCell(props: CellEditorProps) {
  const { cell, isFocused, isSelected, isEditing } = props
  if (!isEditing) {
    return (
      <GridCellDisplay status={cell.status} isFocused={isFocused} isSelected={isSelected}>
        {cell.raw}
      </GridCellDisplay>
    )
  }
  return <MyEditor {...props} /> // mounts only for the edited cell
}
```

### Portaled editors — tag the surface with `data-grid-cell-editor`

If your editor renders its interactive surface in a **portal** — a `Popover`, `Command` dropdown, calendar, time picker, or anything mounted outside the cell's DOM — mark that portaled element with the `data-grid-cell-editor` attribute:

```tsx {5}
<Popover open onOpenChange={(open) => !open && onEditingChange(false)}>
  <PopoverTrigger render={<button />}>{cell.raw || placeholder}</PopoverTrigger>
  {/* The calendar portals out of the cell — tag it so the grid ignores
      pointer events inside it. */}
  <PopoverContent data-grid-cell-editor="">
    <Calendar onSelect={/* onCommit(...) */} />
  </PopoverContent>
</Popover>
```

React events still bubble through the React tree from a portaled child back to the grid, and the grid also watches document-level `mousedown` to clear the selection on click-away. Both paths treat any element inside `[data-grid-cell-editor]` as "still inside the editor": a mousedown there will **not** reselect the cell or `deselect()` — either of which clears `editingCell` and unmounts your editor _before_ its own `onSelect` / click fires, so the pick would silently no-op. The built-in `GridDateCell` and `GridComboboxCell` tag their popovers this way; do the same for any custom portaled editor. Segmented / inline editors that stay inside the cell don't need it.
